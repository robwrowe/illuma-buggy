# E905 Packets - Single Color from Palette

Lights all five LEDs in the same color from the pre-programmed, internal color palette.

```text                                     Palette
                                       ──────────────────
0x8301 E100 E905 00 09 0E ED B0 ────── 13 (Purple)        - On for 14.5 sec
0x8301 E100 E905 00 11 0E F7 B0 ────── 23 (Teal)          - On for 1.5 sec, fade for 0.5 sec
0x8301 E200 E905 00 11 0E E4 B0 ────── 04 (Blue)          - On for 1.5 sec, fade for 0.5 sec
0x8301 E100 E905 00 33 0E E0 BD ────── 00 (Green)         - On for 5 sec, fade for 1.5 sec
0x8301 E200 E905 00 12 0E F5 BB ────── 21 (Red)           - On for 3 sec, fade for 0.5 sec - Vibrate for 1 sec
  ││││ ││││ ││││ ││ ││ ││ ││ ││
  ││││ ││││ ││││ ││ ││ ││ ││ └┴─────── Byte   10 - 0xB0   - Vibration
  ││││ ││││ ││││ ││ ││ ││ └┴────────── Byte   09 - 0xF5   - Color & Mask
  ││││ ││││ ││││ ││ ││ └┴───────────── Byte   08 - 0x0E   - Purpose unknown
  ││││ ││││ ││││ ││ └┴──────────────── Byte   07 - 0x22   - Timing
  ││││ ││││ ││││ └┴─────────────────── Byte   06 - 0x00   - Spacer
  ││││ ││││ └┴┴┴────────────────────── Bytes 4-5 - 0xE905 - OP Code for Effect
  ││││ └┴┴┴─────────────────────────── Bytes 2-3 - 0xE200 - Purpose Unknown
  └┴┴┴──────────────────────────────── Bytes 0-1 - 0x8301 - Disney Identifier
```

## Byte Details

### Bytes 0-1 - Disney Identifier

Always the Disney identifier. Required.

### Bytes 2-3 - Unknown

Purpose unknown. Can be omitted.
Other values: `0xE100`

### Bytes 4-5 - Op Code

OP Code for effect. Required.

### Byte 06 - Spacer

Spacer. Required.

### Byte 07 - Timing

Timing formula. See [timing byte detail](../timing-byte.md) for details.

**Confirmed from lab testing** (superseding earlier documented/assumed values):

- **Fade-out time** is `fade_bits × 0.5s`, not `× 1s` as originally documented. Confirmed
  exact at fade_bits=`01` (0.5s) and fade_bits=`11` (1.5s); fade_bits=`10` (1.0s) is untested
  but fits the pattern.
- **On-time** depends on the scaler bit (bit 6):
  - Scaler = `0`: `on_time ≈ 1.6 × t` (small residuals, ~0.1-0.2s, likely stopwatch precision)
  - Scaler = `1`: `on_time = 3.0 × t` (confirmed exact, zero error across 3 tested points)
- **Bit 7**, documented elsewhere as an "always-on flag," is confirmed to do something real
  but not what the name implies — it does not hold the light on indefinitely. Instead it
  substitutes a third on-time formula, `on_time = 7.6 × t` (tested only with scaler=0 so far),
  producing a much longer but still finite timeout. Fade behaves normally under this mode. See
  [timing byte detail](../timing-byte.md) for the full writeup and worked examples.
- **`t = 0` is a reserved fallback**, not a literal zero — confirmed to produce a flat 3-second
  on-time (tested at scaler=0, fade=0) rather than the ~0s either formula would predict. Not
  yet confirmed whether this fallback holds under other scaler/fade combinations.

### Byte 08 - Unknown.

Purpose unknown. Required.

### Byte 09 - Mask & Color

See [Color and Mask Palette](../color-and-mask-palette.md) for details.

**Confirmed from lab testing:** the split is top 3 bits = mask, bottom 5 bits = color
palette index — verified against all 5 packets shown above, each matching its labeled
palette number exactly (e.g. `0xF5` → mask `111`, color index `21`, matching "Palette 21").

**Also confirmed:** MagicBand+ and Starlight Wand do not always render the same palette
index as the same color — some indices agree between devices (8, 16 confirmed), others
diverge consistently (3, 24 confirmed — same index, different but internally-consistent
color per device). Every one of the 5 packets above uses mask `111` ("all LEDs, single
color"); the other 7 possible mask values are untested.

### Byte 10 - Vibration Pattern

See [EMCOT.world](https://emcot.world/Disney_MagicBand%2B_Bluetooth_Codes#Vibration_Palette) for more information.

## Building Your Own

```text
83 01 <E1|E2> 00 E9 05 00 <TIMING> 0E <MASK+COLOR> <VIBRATION>
```

### `<TIMING>` (byte 07)

Build bit by bit — see [timing byte detail](../timing-byte.md) for the full confirmed model,
caveats, and additional worked examples:

- **Bits [3-0] — time value (`t`)**: `0`-`15`. Feeds whichever on-time formula applies below.
  Avoid `t=0` unless you specifically want the ~3-second fallback behavior — it does not scale
  down to a shorter time.
- **Bits [5-4] — fade-out**: `00`=no fade, `01`=0.5s, `10`=1.0s (untested), `11`=1.5s.
- **Bit [6] — scaler**: `0` selects `on_time ≈ 1.6×t`, `1` selects `on_time = 3.0×t` (exact).
- **Bit [7] — extended-timeout mode**: `0` = normal (use the scaler formula above), `1` =
  `on_time = 7.6×t` instead, confirmed real but not indefinite. Only tested combined with
  scaler=0.

Formula: `timing = (extended << 7) | (scaler << 6) | (fade_bits << 4) | t`

**Worked example — ~8 seconds on, 1-second fade:**
`t ≈ 8/1.6 = 5`, `fade_bits = 10` (1s, untested but predicted — use `01`/0.5s or `11`/1.5s
instead if you want fully-confirmed ground), extended=0, scaler=0 →
`(0<<7)|(0<<6)|(0b10<<4)|0b0101 = 0x25`

**Worked example — ~70 seconds on, 1.5-second fade:**
`t ≈ 70/7.6 ≈ 9`, `fade_bits = 11` (1.5s, confirmed), extended=1, scaler=0 →
`(1<<7)|(0<<6)|(0b11<<4)|0b1001 = 0xB9` — this exact byte has been tested and observed at
69s on, 1.5s fade.

### `<MASK+COLOR>` (byte 09)

`mask_color = (mask << 5) | color_index`

- `mask` (top 3 bits): every tested packet uses `0b111` ("all LEDs"). The other 7 values are
  untested — see [Color and Mask Palette](../color-and-mask-palette.md) before assuming its
  mapping is correct, since it hasn't been independently re-confirmed the way the color/mask
  split above was.
- `color_index` (bottom 5 bits): `0`-`31`, into the palette. Remember MB and SW can render the
  same index differently — check which device you're targeting before assuming a specific
  color will show up as expected.

**Worked example — solid red, all LEDs:** mask=`111`, color index=`21` →
`(0b111 << 5) | 21 = 0xF5` — matches the confirmed Red packet above exactly.

### `<VIBRATION>` (byte 10)

Every tested packet in this doc uses `0xB0`. See the
[EMCOT vibration palette](https://emcot.world/Disney_MagicBand%2B_Bluetooth_Codes#Vibration_Palette)
for the documented pattern table — not yet independently confirmed against lab data. Likely
low priority for the stroller unless it has its own vibration/haptic hardware to drive.
