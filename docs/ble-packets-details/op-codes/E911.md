# E911 Packets — Two-Color Palette Cross-Fade / Pulse

Lights two zones (inner/outer — now confirmed) in independent colors from the internal
palette, animated according to a trailer block that selects between at least two distinct
animation families. A dedicated testing round has now confirmed the timing formula, zone
mapping, and one real animation-toggle byte pair — a substantial upgrade from the first-pass
structural read.

## Byte Details

### Bytes 0-1 - Disney Identifier

Always the Disney identifier. Required.

### Bytes 2-3 - Purpose Unknown

`0xE1` / `0xE2`. Same cross-opcode deferral as documented in the E9 0E doc — not tested in
isolation here either, for the same reasoning (likely opcode-agnostic, better tested once
across several opcodes at once than repeatedly per-opcode).

### Bytes 4-5 - Op Code

`0xE9 0x11`. Required.

### Byte 06 - Spacer

`0x00`. Required.

### Byte 07 - Timing

**Confirmed: `on_time = 1.5 × t` (scaler=0), exact fit across the entire tested range.** A
dedicated sweep from `t=0` to `t=8` (holding colors and trailer fixed on the confirmed
red/blue cross-fade base) produced a **zero-error** match at every tested point from `t=2`
through `t=8`:

| t   | Total time observed | Predicted (1.5×t) | Match |
| --- | ------------------- | ----------------- | ----- |
| 2   | 3.0s                | 3.0s              | exact |
| 3   | 4.5s                | 4.5s              | exact |
| 4   | 6.0s                | 6.0s              | exact |
| 5   | 7.5s                | 7.5s              | exact |
| 6   | 9.0s                | 9.0s              | exact |
| 7   | 10.5s               | 10.5s             | exact |
| 8   | 12.0s               | 12.0s             | exact |

This is the cleanest timing fit found anywhere in this project so far — no residual at all,
unlike every E9 05/09/0E fit, which had small (0.1-0.75s) stopwatch-precision errors. It
exactly matches E9 0E's confirmed scaler=0 formula, reinforcing that this base formula is
shared across opcodes.

**`t=0` and `t=1` weren't given an explicit total**, but their described per-half-cycle timing
(1s transition + hold) is consistent with the same formula at very short total durations too
short to show a clean full-cycle boundary in the notes.

**Internal cycle structure changes somewhere between `t=3` and `t=4`, even though the total
time formula doesn't.** At `t=1-3`, each 1.5s "unit" of the cycle was described as a 1-second
color transition followed by a separate 0.5-second hold at the new color, then a hard cut back
to the start color. At `t=4-8`, the transition itself was described as taking the full 1.5s
with no separate hold mentioned, before cutting back. Both structures total 1.5s per cycle
unit and both fit the same overall formula — this is a real, noted behavioral difference in
how that 1.5s is spent, not a formula break. Not yet explained; flagged as an open detail
rather than investigated further.

**Confirmed: `on_time = 3.0 × t` (scaler=1) also holds.** The timing byte `0x54` (used as the
default base for most of this round's tests) decodes to `scaler=1, t=4, fade_bits=01`. Its
observed behavior — "On for 12.5s, fade out for ~0.5s" — matches `3.0×4=12.0s` almost exactly,
with the extra ~0.5s plausibly the separately-timed fade tail (`fade_bits=01` predicting a
0.5s fade, which also matches here — unlike E9 0E, where this same fade formula was found to
be broken). Both E9 05/09/0E's scaler formulas now transfer cleanly to E9 11, including fade
timing, which is a stronger transfer than E9 0E managed.

### Byte 08 - Marker

`0x0F` in every sample.

### Byte 09 - Inner Zone: Color & Mask — CONFIRMED

`mask_color = (mask << 5) | color_index`, top-3-bits read as `010` (not `101` as in E9 05/09/
0E — doesn't appear to affect color, see color palette notes elsewhere in this project). **A
direct test with two clearly distinct colors (red at byte 9, cyan at byte 10) resulted in
"Cyan Outer, Red Inner" — confirming byte 9 is the inner zone.** Cross-checked successfully
against a second, independent test (the bytes-13/14 sweep, which used byte9=blue/byte10=red
and was independently described as "Inner blue, Outer Red") — two-for-two agreement.

### Byte 10 - Outer Zone: Color & Mask — CONFIRMED

Same color formula as byte 09. Confirmed outer zone per the test described above.

### Bytes 11, 12, 15, 16, 19, 20 - Animation-Family Selector (theory, swap test inconclusive)

These six bytes still move in lockstep across every sample in the dataset (two value-sets,
"Mode A" = pulse-family, "Mode B" = cross-fade-family — see the original bootstrap doc for the
full breakdown). **A direct test attempting to convert a Mode-B (cross-fade) packet into
Mode-A (pulse) by swapping only these 6 bytes did not produce a pulse effect — the device did
not respond at all, tested 3 times.** This doesn't confirm or refute the underlying theory: it
shows that these 6 bytes can't simply be transplanted onto an otherwise-unrelated payload
without something else being satisfied too — plausibly there's an additional
consistency/validation requirement (in the spirit of E9 0E's fixed byte 18) linking this block
to other parts of the packet that wasn't accounted for in the swap. **The lockstep correlation
itself is still real and unexplained by this result** — it just means "cut and paste these 6
bytes" isn't the whole story. Worth retrying with a more conservative swap (e.g. also using a
Mode-A base packet's own colors/timing, not just the trailer) before concluding anything
further.

### Bytes 13, 14 - Cross-Fade Animation ON/OFF Toggle — CONFIRMED

**A real, newly-discovered parameter pair, not previously understood.** Three different value
pairs (`00/00`, `FF/FF`, `20/80`) were tested against the confirmed-working baseline (`48/82`),
holding everything else fixed on the red/blue cross-fade packet. **All three non-baseline
values produced the identical result: the effect collapsed from a smooth cross-fade into a
static solid-on display** — no animation at all, just the two zones showing their assigned
colors continuously (inner blue, outer red, matching the confirmed zone mapping exactly), with
the same overall on-time and fade-out timing as the animated version. This is a clean,
all-or-nothing toggle, not a graduated parameter — every tested non-baseline value gave the
same solid-on result regardless of what it was, similar in spirit to E9 0E's bytes 14/17
(where any non-baseline value also produced one consistent fallback behavior). **Practical
takeaway: bytes 13-14 must be exactly `48 82` for the cross-fade motion to actually play; any
other tested value freezes it into a static two-color display instead.**

### Bytes 17, 18, 21 - Cross-Fade Sub-Cluster — CONFIRMED, and corrected

Within the cross-fade family, a sub-cluster of these three bytes taking the values `05 / 0F /
00` was previously theorized to separate "clean" from "chaotic" cross-fades. **A direct swap
test confirms these bytes are real and do change the sub-effect — but the result was a
different, better-characterized effect than "chaotic."** Swapping bytes 17/18/21 from the
clean values (`05/0F/00`) to the previously-"ambiguous" values (`06/09/43`) on the confirmed
red/blue cross-fade packet produced: **each zone alternating color and strobing at
different, but repeatable intervals — the same pattern every time it played, explicitly not
random.** This corrects the earlier "possibly random" framing from the bootstrap doc — the
sub-effect selected by non-baseline values here is a real, deterministic, different animation
(a region-alternating strobe), not noise or instability. The original "may be three colors" /
"random pulse" language in some of the untested bootstrap samples may describe yet other,
still-uncharacterized values within this same byte trio, rather than genuine randomness —
worth re-examining those specific samples with this corrected understanding in mind.

### Byte 22 - Vibration (theory, unconfirmed)

**Inconclusive, not confirmed.** A sweep of `0x00`, `0xB0` (baseline), and `0xFF` all produced
identical results, and all three were explicitly noted as producing **no vibration at all** —
including at `0xFF`, which should plausibly be a strong pattern if this byte works the way
E9 0E's vibration byte does. Two explanations are equally consistent with this result: either
this byte genuinely doesn't control vibration for this opcode (or does nothing at all), or the
physical test device's vibration motor wasn't functioning/felt during this test session. Given
E9 0E's vibration byte was confirmed independent-but-functional using the same kind of sweep,
the fact that _no_ value produced _any_ vibration here (not even a difference between values)
is a meaningfully different result from E9 0E's, and worth a deliberate retest — ideally
holding the wand directly and confirming the test device's vibration motor works at all with a
packet from an opcode already known to vibrate, before concluding anything about this byte.

## Building Your Own

```text
83 01 <E1|E2> 00 E9 11 00 <TIMING> 0F <INNER_COLOR> <OUTER_COLOR> <11-byte trailer> <VIBRATION>
```

### `<TIMING>` (Byte 07) — formula confirmed

```
on_time = 1.5 × t     (scaler = 0, confirmed exact across t=2-8)
on_time = 3.0 × t     (scaler = 1, confirmed via t=4 example)
```

`byte = (scaler << 6) | (fade_bits << 4) | t`. The always-on/extended-timeout flag (bit 7)
has not been tested for this opcode — leave at `0` until confirmed.

**Worked example — a 9-second cross-fade cycle:** `t=6, scaler=0` → `byte = (0<<6)|(0<<4)|0b0110
= 0x06` — this exact byte has been tested and confirmed at 9.0s total, 6 cycles.

### `<INNER_COLOR>` / `<OUTER_COLOR>` (Bytes 09-10) — zone mapping confirmed

Byte 09 = inner zone, byte 10 = outer zone. `mask_color = (0x40 | color_index)` — top-3-bits
`010` convention observed for this opcode; `0xA0 | index` would very likely also work given
color has never depended on the top 3 bits anywhere else in this project, but untested.

### Bytes 13-14 — must stay at `48 82` for animation to play

**Confirmed required for the cross-fade motion itself.** Any other tested value collapses the
effect into a static, non-animated two-color display. If you want a static two-tone look
rather than a cross-fade, this is actually a usable, confirmed way to get one — just don't
expect motion unless these two bytes are exactly `48 82`.

### Confirmed working examples

**Standard cross-fade, adjustable timing** (swap the timing byte for any confirmed `t` value
above; swap bytes 09-10 for any other two colors):

```
e200e91100540f445558f44882d146050fd06500b0
```

**Region-alternating strobe variant** (same colors/timing, different sub-cluster bytes):

```
e200e91100540f445558f44882d1460609d06543b0
```

**Static two-tone (no animation)** — any of these three tested byte13/14 pairs works
identically:

```

e200e91100540f445558f4ffffd146050fd06500b0
e200e91100540f445558f42080d146050fd06500b0
```

## Suggested next testing round

1. **Retry the animation-family swap with a more complete transplant.** Rather than only
   swapping bytes 11/12/15/16/19/20, try copying an entire confirmed-working Mode-A packet's
   colors and timing byte alongside its trailer, to rule out an interaction with the
   unchanged parts of the payload as the reason the previous attempt failed to respond.
2. **Re-examine the remaining "ambiguous" bootstrap samples** (the ones with 3-color/random
   language) against the corrected sub-cluster understanding — check whether their bytes
   17/18/21 values form their own distinct, describable sub-effects rather than genuine
   randomness, the way `06/09/43` turned out to be a real region-alternating strobe rather
   than chaos.
3. **Confirm or rule out the vibration byte with a sanity check first** — test a
   known-to-vibrate packet from another opcode on the same physical device before retesting
   byte 22 here, to separate "this byte doesn't work" from "the test device's vibration motor
   isn't functioning."
4. **Characterize the `t=3`/`t=4` cycle-structure transition** noted in Byte 07 — is it a hard
   switch at a specific `t` value, or a gradual shift? Test `t=3` and `t=4` a few more times
   each, focusing specifically on the transition/hold breakdown within each cycle rather than
   just total duration.
5. **Test the scaler=1 formula more directly** — only one data point (`t=4`) exists so far.
   A proper sweep (e.g. `t=1,2,3` at scaler=1) would confirm the formula the way scaler=0's was.
