# E911 Packets - Base + Chase / Paired-Alternation Color (Not Yet Implemented)

**No rule currently exists for this opcode in the rules engine.** This doc captures the
working theory from lab observation and byte-isolation testing, for use when building a rule.

Part of a related family with [E9 10](./e910.md), [E9 12](./e912.md), [E9 13](./e913.md), and
[E9 14](./e914.md) — payloads of increasing length that share timing-byte position, a `0F`
marker, and a growing block of color bytes (2 for E9 10, 3 for E9 11, 4 for E9 12, 5 for E9
13, 5+extra for E9 14). Structurally these look like one shared format parameterized by zone
count rather than five unrelated opcodes — see the
[rules engine reference](./rules-engine-reference.md) for the cross-opcode alignment that
established this.

timing byte, consistent with how the related opcodes' extract offsets have been read during
lab analysis (not yet formalized into a rule's own numbering).

## Observed Behavior

Lab notes for this opcode consistently describe cross-fading, pulsing, or ping-ponging between
**inner** and **outer** (sometimes a third color), matching the same inner/outer conceptual
model as [E9 0F](./e90f.md) rather than the five-independent-corner model used by E9 09/E9
0C/E9 0D/E9 0E.

**Two distinct color-encoding sub-patterns have been observed within this opcode's 3 zone
bytes:**

1. **Base + chase.** One byte is a static/dim "base" color, another is an independently
   varying "chase/vivid" color — confirmed via controlled byte-isolation testing on the
   related [E9 13](./e913.md) opcode (same family, 5 zone bytes instead of 3), where
   overriding byte 0 changed the "dim" color and byte 1 changed the "vivid/chase" color
   independently. Not yet independently isolated for E9 11 specifically — carried over as a
   working assumption based on the shared family structure.
2. **Paired-alternation.** In some E9 13 samples, zone bytes showed a repeating pattern where
   byte 1 == byte 3 and byte 2 == byte 4 (two alternating pairs sharing state), rather than N
   independent values — see [E9 13](./e913.md) for the fullest write-up of this pattern, since
   it was most clearly isolated there. With only 3 zone bytes, E9 11 doesn't have enough bytes
   to exhibit the same 2-pair structure directly, so this sub-pattern may not apply to this
   specific opcode the same way.

Which of these two patterns (or a third, opcode-specific one) actually governs E9 11's 3 zone
bytes has not been isolated with a dedicated test — treat both as candidate theories pending
confirmation.

## Timing

Standard timing-byte bit layout applies at the byte position immediately preceding the `0F`
marker — confirmed via an E9 13 sweep in this family (`05`→~8s/0.5s cycle, `0F`→~24s/0.5s
cycle, `30`→~1.5s/1s cycle, `B0`→~7.5s/1s cycle, matching `timing-byte.md`'s scaler-bit and
time-value behavior directly). Not independently re-confirmed for E9 11 specifically, but
given the shared family structure this is a reasonably strong working assumption — stronger
than the color-byte theories above, which haven't been cross-validated at all for this opcode.

## Building Your Own

Not yet buildable with high confidence — recommend resolving the base+chase vs.
paired-alternation question (or determining E9 11 uses a third pattern) before building a
rule. If experimenting, copy a working sample's trailer verbatim and vary only the timing byte
and the 3 zone-color bytes.

## Suggested Lab Tests

1. **Byte-role isolation** — same methodology already run successfully on E9 13: hold trailer
   and timing constant, override one zone byte at a time to a distinct, identifiable color,
   and observe which visual role (base/dim vs. chase/vivid, or a fixed position) each byte
   controls.
2. **Cross-check against E9 13's pattern** — once isolated, compare E9 11's actual behavior
   against both candidate patterns above to determine which (if either) generalizes across the
   family, or whether each opcode in the E9 10-E9 14 family needs its own independent theory.
3. **Timing confirmation** — run the same timing sweep methodology already used on E9 13
   (vary the timing byte, hold colors constant, measure total on-time and cycle time) to
   confirm the shared timing model applies here too.
