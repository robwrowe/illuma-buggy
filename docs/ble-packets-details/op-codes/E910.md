# E910 Packets - Base + Chase / Inner-Outer Color, 2-Zone (Not Yet Implemented)

**No rule currently exists for this opcode in the rules engine.** This doc captures the
working theory from lab observation, for use when building a rule.

Smallest/first member of the [E9 11](./e911.md)-[E9 14](./e914.md) family — 2 zone-color
bytes, the fewest in the family (E9 11 has 3, E9 12 has 4, E9 13 has 5, E9 14 has 5+extra).
See [E9 13's doc](./e913.md) for the most directly-confirmed findings in this family; E9 10's
theory below is a working assumption carried over from that stronger evidence, not
independently confirmed.

Byte numbers left blank — no shipped rule exists yet. As with the rest of this family, a
second payload branch without the `0F` marker has also been observed (second sample above
starts `01 0F 5C 55 F4 A0 D0 37...` — note this one *does* have `0F` at the marker position,
unlike E9 14's non-`0F` samples; the family's branch-vs-single-format situation isn't
consistent across every member and should be checked per-opcode rather than assumed).

## Observed Behavior

Notes consistently describe two-color inner/outer behavior — cross-fading, flickering, or a
single cut-on-then-fade-out — consistent with the family's general inner/outer conceptual
model. One sample ("cycles through the rainbow, all 5 LEDs same color") describes behavior
that doesn't fit the 2-zone inner/outer model at all, suggesting either a distinct sub-mode
selected by a byte not yet identified, or that this specific sample belongs to a structurally
different branch than the rest.

## Color-Byte Structure (working assumption, unconfirmed for this opcode specifically)

With only 2 zone-color bytes, E9 10 doesn't have enough bytes to exhibit E9 13's 4-byte
paired-alternation pattern — if a "base + chase" style relationship applies here at all
(carried over from [E9 13](./e913.md#pattern-a--base--chase), not confirmed for E9 10), it
would most likely mean the 2 bytes are simply two independent colors (inner, outer) rather
than one base + one moving highlight, since there's no room for a "chase" to move between
positions with only 2 zone bytes. This is a plausible simplification of the family pattern for
the shortest member, but hasn't been tested.

## Timing

Standard family timing model assumed (see [E9 13](./e913.md#confirmed-timing-behavior) for the
directly-confirmed version) — not independently tested for E9 10.

## Building Your Own

Not yet buildable with confidence. If experimenting, treat the 2 zone-color bytes as
independent inner/outer values (rather than base+chase, given the byte-count reasoning above)
as a starting hypothesis, and copy a working sample's trailer verbatim.

## Suggested Lab Tests

1. **Zone-byte role isolation** — hold one byte constant, override the other to a distinct
   color, confirm independent inner/outer control (or find evidence of a base+chase
   relationship instead).
2. **"Rainbow cycle" sample follow-up** — determine whether this describes a genuinely
   different sub-mode/branch, and if so what byte selects it.
3. **Timing confirmation** — run the same sweep methodology already used on E9 13.
4. **Marker-byte consistency check** — confirm whether all E9 10 samples share the `0F` marker
   at the same position, or whether (like E9 14) some samples use a structurally different
   payload shape entirely.
