# E90C Packets - Multi-Zone Color (Solid / Chase)

Lights the five LED zones (Center, TopRight, BottomRight, BottomLeft, TopLeft) using either a
single shared color with a solid/strobe timing behavior, or five independently-assigned colors
that animate as a chase (UL → BL → BR → UR → Center). Which behavior occurs is selected by
byte 14.

```text Sub-mode
                                                          ──────────────────
0x8301 E100 E90C 00 64 0F B2 A4 B5 B2 B5 30 7B 7D B0 ────── Chase     - red x2, yellow x2, blue x1
0x8301 E100 E90C 00 CF 0F B9 A4 B9 A4 A8 30 7B 7B B0 ────── Chase     - green x2, blue x2, purple x1
0x8301 E100 E90C 00 01 0F 5C 5C 5B F0 00 FC 37 48 95 ────── Solid     - white, ~1.5s on
0x8301 E100 E90C 00 EF 0F 4F 4F 5B F0 FB 14 37 48 95 ────── Solid     - orange, slow strobe

  ││││ ││││ ││││ ││ ││ ││ ││ ││ ││ ││ ││ ││ ││ ││ └┴─────── Byte   18 - Vibration
```

Note: raw lab captures for this opcode typically start at the OP code (`E9 0C ...`) without the
`8301 <E1|E2> 00` prefix. That prefix is confirmed present and required in practice, per the
[E9 05](./e9-05.md) documentation — the diagram and byte numbering above include it for
consistency with that doc. All byte-detail sections below reference these full-frame indices.

## Byte Details

### Byte 07 - Timing

Same bit layout as [timing byte detail](../timing-byte.md), and confirmed to match the
established on-time formula in solid sub-mode:

- `0x01` (t=1, scaler=0, fade=0) → predicted on-time `1.6×1=1.6s`, matches observed
  "Solid white for 1.5s" closely.

**Unconfirmed:** in chase sub-mode, this byte does not appear to map cleanly to a single
on/fade cycle the way it does in solid mode — observed chase samples describe multi-second
"total time" and "cycle time" figures that don't resolve neatly against the standard formula
(e.g. `0x64` → t=4, scaler=1 → predicted `3.0×4=12s`, vs. an observed ~14s "on time" — close
but not exact, and it's unclear whether the byte here means total duration, per-step dwell, or
something else). Needs a dedicated timing sweep in chase mode to resolve.

### Byte 08 - Length/Marker

Constant `0x0F` across every sample collected so far (11/11). Likely a fixed marker or length
byte, consistent with the same constant appearing in E9 0E. Not yet tested with other values.

### Bytes 09-13 - Per-Zone Color

Five consecutive bytes, one per LED zone, in the order **TopLeft, BottomLeft, BottomRight,
TopRight, Center** — same zone ordering already confirmed for E9 09.

**Confirmed:** each byte follows the same mask+color split as
[E9 05 byte 09](./e9-05.md#byte-09---mask--color) (top 3 bits = mask, bottom 5 bits = color
index into the [Color and Mask Palette](../color-and-mask-palette.md)).

**Confirmed anomaly:** every zone byte across every sample (both sub-modes) decodes to
**mask = `0b101` (5)**. Per the palette doc, mask `101` nominally means "All LEDs" — but that
reading doesn't make sense for a per-zone byte where each zone independently carries a
different color. Working theory: mask bits are not meaningful in this opcode and `0b101` is a
fixed/reserved value baked into every zone byte here — i.e. only the bottom 5 (color) bits
should be treated as variable. Not yet tested with any other mask value; unknown whether
setting a different mask value here has any effect or is even accepted by the device.

**Confirmed from lab data (chase sub-mode):** color-index decode cross-checked against repeat
counts and named colors in observation notes across 4 independent samples — all matched:

| Zone bytes (TL BL BR TR C) | Decoded colors                                                              | Observed note                  |
| -------------------------- | --------------------------------------------------------------------------- | ------------------------------ |
| `B2 A4 B5 B2 B5`           | chartreuse×2, vivid-blue×1, red-orange×2                                    | "red x2, yellow x2, blue x1"   |
| `B9 A4 B9 A4 A8`           | lime-green×2, vivid-blue×2, purple-pink×1                                   | "green x2, blue x2, purple x1" |
| `B1 B9 B5 B1 A2`           | strong-yellow×2, lime-green×1, red-orange×1, royal-blue×1                   | "yellowx2, red, green, cyan"   |
| `A9 B9 B5 B1 A2`           | bright-pink, lime-green, red-orange, strong-yellow, royal-blue (5 distinct) | 5-color chase across zones     |

As with E9 05, expect MagicBand+ and Starlight Wand to render the same color index
differently — the perceived color names in the notes (e.g. "red" for `red-orange`, "purple"
for `bright-pink`) are consistent with known cross-device divergence, not decode errors.

**Solid sub-mode:** all 5 zone bytes typically share the same color value in the two solid
samples collected (`5D 46 5B F0 ...` type patterns show near-identical or matching values),
consistent with "solid color, all LEDs."

### Byte 14 - Sub-mode Selector

- `0x30` → **Chase mode**: the 5 zone-color bytes are treated as independent per-zone colors
  and the effect animates as a chase in TL→BL→BR→TR→Center order. 4/4 samples with this value
  matched chase behavior in notes.
- Other observed values (`0xF0`) → **Solid/strobe mode**: zone-color bytes behave as a single
  shared color (or the animation is a fade/strobe rather than a positional chase). 7/7 samples
  with `0xF0` matched solid/strobe/blink-style notes.

Only two distinct values observed (`0x30`, `0xF0`) — unconfirmed whether this is a true
bitfield with other valid selector values, or simply a two-state flag. Treat as an opaque
enum until more values are captured.

### Bytes 15-17 - Tail

Purpose unknown. Values vary across samples (`37 48 95`, `30 7B 7D` + trailing byte, etc.) and
don't yet correlate cleanly with any observed behavior. Possibly checksum, additional
timing/fade parameters, or animation-speed control for chase mode. Needs isolated sweep
testing — hold everything else constant and vary one tail byte at a time.

## Building Your Own

```text
83 01 <E1|E2> 00 E9 0C 00 <TIMING> 0F <ZONE_TL> <ZONE_BL> <ZONE_BR> <ZONE_TR> <ZONE_C> <SUBMODE> <TAIL x3>
```

### `<TIMING>` (byte 07)

Same formula as [E9 05](./e9-05.md#timing-byte-07). Confirmed reliable for solid sub-mode
only; treat as unverified for chase sub-mode until a dedicated timing sweep is run.

### `<ZONE_*>` (bytes 09-13)

`zone_byte = (0b101 << 5) | color_index` — i.e. always OR your color index (0-31) with `0xA0`
base value, per the mask=101 anomaly noted above.

**Worked example — TopLeft = vivid-blue (index 4):** `(0b101 << 5) | 4 = 0xA4` — matches
confirmed samples above exactly.

For chase mode, assign five different `color_index` values across the five zone bytes. For
solid mode, use the same `color_index` in all five (or at minimum in however many zones you
want lit — untested with mixed values in solid sub-mode).

### `<SUBMODE>` (byte 14)

`0x30` for chase, `0xF0` for solid/strobe. Other values unconfirmed.

### `<TAIL>` (bytes 15-17)

Unconfirmed. Copy verbatim from a working captured sample of the sub-mode you're targeting
until isolated testing resolves these bytes.

## Chase `sx` calibration (WLED fx=28)

`fx=28` is WLED `FX_MODE_CHASE_COLOR` ("Chase") in upstream `wled00/FX.h` / `FX.cpp`.

Chase timing (not Strobe’s `255 - 50/hz`):

```text
counter = strip.now * ((sx >> 2) + 1)
a       = (counter * SEGLEN) >> 16
T_ms    = 65536 / ((sx >> 2) + 1)     // one full lap, independent of SEGLEN
sx      = clamp(round(4 * (65536/T_ms - 1)), 0, 255)
```

For E9 0C chase, set **T = Disney on_time** (`1.5×tval` scaler=0, `3.0×tval` scaler=1) so one lap
matches confirmed wand duration; per-zone dwell ≈ `on_time/5`.

Helpers: `chaseSxFromCycleMs` / `e90cChaseSpeedBuckets` in `web/src/lib/ble/e9Decode.js`,
and `chaseSxFromCycleMs` in `MbRuleEngine.cpp`.

WLED floor: slowest Chase ≈ **65.5s** (`sx≤3`). Scaler=1 / high tval cannot match Disney’s longer
on-times exactly — use the table’s best available `sx`. Reconfirm effect name via GLEDOPTO
`GET /json/eff` index 28 when on `StrollerNet`.

Use per-tval `speedBuckets` (`maxByte` 1…15, `maskBits` tval nibble) rather than 4 coarse bands.

1. **Zone-color isolation** — hold 4 zones at a known value (`A4`, vivid-blue) and sweep the
   5th zone through candidate color indices one at a time, in chase mode (`SUBMODE=0x30`).
   Confirms exact zone-to-position mapping and fills in unidentified color indices (`B9`, `B5`,
   `A2`, `A9`, etc.) with certainty rather than inference.
2. **Mask-bit sweep** — hold color index constant, vary the top 3 bits of one zone byte across
   all 8 mask values, chase mode. Confirms or refutes the "mask is fixed/ignored" theory.
3. **Chase timing sweep** — hold zone colors and submode constant, sweep byte 7 (timing) across
   a range of `t` values at both scaler settings. Stopwatch total-time and cycle-time for each,
   per the documented jitter-aware measurement baseline (start timing from first observed
   effect onset).
4. **Tail-byte sweep** — hold everything else constant, vary bytes 15-17 one at a time in both
   sub-modes, watching for changes in animation speed, fade behavior, or repeat count.
