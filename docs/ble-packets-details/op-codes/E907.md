# E907 Packets - Single Color (RGB)

Lights all five LEDs in the same color, specified as a raw RGB triple rather than an internal
palette index. A second, less-understood payload shape also exists under this opcode — see
below.

```text                                              R    G    B
                                                    ──────────────
0x8301 E100 E907 00 21 D2 55 FC 64 00 ────────────── 252  100  0   - vibrant yellow/orange
0x8301 E100 E907 00 24 D2 55 FC 00 00 ────────────── 252  0    0   - bright red
0x8301 E100 E907 00 4F D2 55 00 41 FC ────────────── 0    65   252 - cyan/teal
0x8301 E100 E907 00 4F D2 55 3D 00 FC ────────────── 61   0    252 - light purple
0x8301 E100 E907 00 74 D2 55 00 40 04 ────────────── 0    64   4   - vivid green
  ││││ ││││ ││││ ││ ││ ││ ││ ││ ││ ││ ││
  ││││ ││││ ││││ ││ ││ ││ ││ ││ ││ └┴─────────────── Byte   10 - Blue channel (0-255)
  ││││ ││││ ││││ ││ ││ ││ ││ ││ └┴──────────────────  Byte   09 - Green channel (0-255)
  ││││ ││││ ││││ ││ ││ ││ ││ └┴─────────────────────  Byte   08 - Red channel (0-255)
  ││││ ││││ ││││ ││ ││ ││ └┴────────────────────────  Byte   07 - 0xD2 marker (RGB-mode selector, byte 1 of 2)
  ││││ ││││ ││││ ││ ││ └┴───────────────────────────  Byte   06 - 0x55 marker (RGB-mode selector, byte 2 of 2)
  ││││ ││││ ││││ ││
  ││││ ││││ ││││ └┴──────────────────────────────────  Byte   05 - Timing
  ││││ ││││ ││││
  ││││ ││││ └┴┴┴─────────────────────────────────────  Bytes 4-5 - OP Code for Effect (E9 07)
  ││││ └┴┴┴──────────────────────────────────────────  Bytes 2-3 - Purpose Unknown (E1 or E2)
  └┴┴┴───────────────────────────────────────────────  Bytes 0-1 - Disney Identifier
```

**Byte-offset note:** the diagram above numbers bytes relative to the shipped rule's own
extract offsets (`offset: 8/9/10` for R/G/B, `channelGroup`), which are given relative to the
payload immediately following the `E907` opcode bytes. This differs from the full-frame
numbering (including the `8301 <E1|E2> 00` prefix) used in E9 05/E9 09/E9 0C's docs — treat
byte numbers here as the shipped rule's own convention, not independently re-derived against
those other docs' numbering. If you're cross-referencing offsets between opcode docs, confirm
which convention each one uses before assuming they line up.

A second, structurally different payload shape has also been observed under this opcode and
is **not** matched or handled by the shipped rule:

```text
0x8301 E200 E907 00 22 0F 44 44 41 B0 ── vivid blue
0x8301 E200 E907 00 22 0F 43 43 41 B0 ── light blue / purple
```

This branch uses a `0F` marker (matching the zone-color family's marker byte) followed by two
repeated single bytes and a trailer, rather than the `D2 55` RGB marker. Only two samples
exist, both at the same timing byte (`0x22`), and the bytes don't decode cleanly against the
standard mask+color split (`0x44` → mask=`0b1000`(8), color=`0b00100`(4) under that scheme,
which doesn't correspond to "vivid blue" in the [Color and Mask Palette](../color-and-mask-palette.md)).
Treat this branch as unresolved — needs more samples at varied timing/byte values before a
real theory can be built. Packets in this shape currently fall through to lower-priority rules
or the unhandled-opcode fallback, since `e9-07_solid_rgb` only matches the `D2 55` branch.

## Byte Details

### Byte 05 - Timing

Same bit layout as [timing byte detail](../timing-byte.md). The shipped rule
(`e9-07_solid_rgb`) does not set a dedicated `timingModelId` (uses the rules engine's default
timing resolution) — this has not been independently confirmed against E9 07-specific timing
sweeps. Given E9 05/E9 09 share one timing model and E9 0E turned out to need its own, don't
assume E9 07 necessarily matches the default without testing.

### Bytes 06-07 - `D2 55` RGB-Mode Marker

Two fixed bytes that select the raw-RGB payload format, as opposed to the `0F` marker used by
the palette-indexed zone-color family (E9 09/E9 0C/E9 0D/E9 0E) and by this opcode's own
second, unresolved branch (see above). Confirmed via 6+ color-labeled lab samples decoding
cleanly as direct 8-bit-per-channel RGB (0-255 range) — notably *not* limited to the 0-31
range a palette index would use, which is the strongest evidence this is genuinely raw color
data rather than an index into the [Color and Mask Palette](../color-and-mask-palette.md).

This same `D2 55` marker (and a second `D2 58` marker for a two-block variant) appears across
multiple opcodes — see [E9 08](./e908.md) and [E9 0D](./e90d.md) for the shared-format
writeup. E9 07 is the opcode where this format was first confirmed.

### Bytes 08-10 - RGB Channels

The shipped rule extracts these via a `channelGroup` with `scale: "bitReplicate6to8"` and
`bitStart: 1, bitCount: 6` — i.e. **currently configured as if these were 6-bit-packed
values**, the same scheme used for the older E9 xx palette-adjacent formats. This does not
match the confirmed lab findings below, which show these bytes as **direct 8-bit values** (R
values of 252, 61, etc. — well outside a 6-bit range's maximum of 63). **This looks like a
real discrepancy between the shipped rule's extract configuration and the confirmed decode** —
worth switching the rule's `channelGroup.scale` to `"direct8"` (already a supported scale mode
in the rules engine) rather than `bitReplicate6to8`, and re-validating against the samples
below before trusting the rule's current output.

Confirmed direct-8-bit values against labeled lab samples:

| Bytes | R | G | B | Observed label |
| ----- | --- | --- | --- | --- |
| `FC 64 00` | 252 | 100 | 0 | vibrant yellow/orange |
| `FC 00 00` | 252 | 0 | 0 | bright red |
| `00 41 FC` | 0 | 65 | 252 | cyan/teal |
| `3D 00 FC` | 61 | 0 | 252 | light purple |
| `00 40 04` | 0 | 64 | 4 | vivid green |
| `4B 00 00` / `57 00 00` | 75 / 87 | 0 | 0 | red (dim) |

A value of `00 00 00` renders as off/black — this is expected raw-RGB behavior (black is a
valid color, not a decode failure), and resolves an earlier open question about samples that
appeared to "go to black" under a different (incorrect) decode assumption.

## Building Your Own

```text
83 01 <E1|E2> 00 E9 07 00 <TIMING> D2 55 <R> <G> <B>
```

### `<TIMING>` (byte 05)

Standard timing-byte formula — see [timing byte detail](../timing-byte.md). Not independently
confirmed for E9 07-specific on-time/fade behavior; treat as a working assumption until tested
directly.

### `<R> <G> <B>` (bytes 08-10)

Direct values, `0`-`255` each, no palette lookup, no bit-packing. **If hand-building a packet
to match the confirmed lab table above, use the raw decimal/hex byte values directly — do not
apply any 6-to-8-bit expansion**, despite what the currently-shipped rule's extract
configuration does on the decode side (see discrepancy noted above). Unlike every
palette-indexed opcode in this project, MagicBand+ vs. Starlight Wand color-rendering
divergence should not apply here, since there's no shared internal palette table being
indexed. Not independently confirmed for Starlight Wand, since all lab samples collected so
far are MagicBand+ only.

## Suggested Lab Tests

1. **Fix and re-validate the shipped rule's scale mode** — switch `e9-07_solid_rgb`'s
   `channelGroup.scale` from `bitReplicate6to8` to `direct8` and confirm output against the
   confirmed lab table above; this looks like a real bug in the current configuration, not
   just an area needing more data.
2. **`0F`-branch investigation** — gather more samples of the second payload shape (`0F 44 44
   41 B0` type) at varied timing bytes, to determine whether it's a distinct sub-format worth
   its own rule, or a malformed/legacy variant not worth pursuing.
3. **Starlight Wand RGB confirmation** — confirm the same `D2 55 <R> <G> <B>` format and
   decode holds for Starlight Wand captures of this opcode.
4. **Timing sweep** — no E9 07-specific timing sweep has been run; confirm the standard
   formula's constants apply here rather than assuming.
