# E904 Packets - Single Color from Palette (Short Form)

Lights all five LEDs in the same color from the internal palette. Structurally the shortest
member of the single-color family — same timing byte and palette-index scheme as
[E9 05](./e9-05.md), but without E9 05's byte 08 (purpose-unknown spacer) or byte 10
(vibration) fields.

```text
0x8301 E100 E904 00 0F 0E F5 ────── On for ~24 sec (t=15, scaler=0)
0x8301 E100 E904 00 12 0E F9 ────── (t=2, scaler=1)
0x8301 E100 E904 00 43 0E FB ────── (t=3, scaler=1, fade=1)
  ││││ ││││ ││││ ││ ││ ││ ││
  ││││ ││││ ││││ ││ ││ ││ └┴─────── Byte   09 - Color & Mask (or trailer — see below, unconfirmed)
  ││││ ││││ ││││ ││ ││ └┴────────── Byte   08 - 0x0E   - Purpose unknown (matches E9 05 byte 08)
  ││││ ││││ ││││ ││ └┴───────────── Byte   07 - Timing
  ││││ ││││ ││││ ││
  ││││ ││││ ││││ └┴────────────────Byte   06 - 0x00   - Spacer
  ││││ ││││ ││││
  ││││ ││││ └┴┴┴───────────────────Bytes 4-5 - 0xE904 - OP Code for Effect
  ││││ └┴┴┴────────────────────────Bytes 2-3 - Purpose Unknown
  └┴┴┴──────────────────────────────Bytes 0-1 - 0x8301 - Disney Identifier
```

**Byte 09's role is not independently confirmed for this opcode.** The currently-shipped rule
(`e9-04_solid`) extracts a full 8-bit palette index at offset 7 with `paletteMap: true` —
**this offset does not match the byte-9 mask+color position used by E9 05/E9 09/E9 0C's
family.** Either (a) E9 04 genuinely places its color byte at offset 7 rather than 9, making
it structurally different from E9 05 despite the similar timing/palette scheme, or (b) the
shipped rule's offset is a carried-over guess that hasn't been validated against a live
device. Given the sample bytes above are only 4 bytes long after the timing byte (`0E F5` /
`0E F9` / `0E FB`), there may not even be a byte 9 in some captures — treat this opcode's
color-byte position as **unconfirmed** until checked against a live test, despite the shipped
rule already extracting *something* from offset 7. This is flagged explicitly rather than
presented as settled, since the offset-7-vs-9 discrepancy hasn't been resolved.

## Byte Details

### Bytes 0-1 - Disney Identifier

Always the Disney identifier. Required.

### Bytes 2-3 - Unknown

Purpose unknown. Same as every other E9xx opcode — can be `E100` or `E200`.

### Bytes 4-5 - Op Code

OP Code for effect. Required.

### Byte 06 - Spacer

Spacer. Required, matches every other E9xx opcode.

### Byte 07 - Timing

Same bit layout as [timing byte detail](../timing-byte.md). The shipped rule doesn't specify a
dedicated timing model (`timingModelId: ""`, meaning the rules engine's default model applies)
— this has not been independently confirmed against E9 04-specific timing sweeps. Given the
strong structural similarity to E9 05, the default (E9 05/E9 09) model is a reasonable working
assumption, but treat predicted on-times as unconfirmed until tested.

### Byte 08 - Unknown

Purpose unknown, matches E9 05's byte 08 position exactly. Constant `0x0E` across all observed
samples (same value E9 05 uses here too).

### Byte 09 (or possibly byte 07) - Color

See the callout above — genuinely unresolved which offset carries the color value, despite the
rules engine currently extracting from offset 7. If following the shipped rule's assumption,
treat this the same as [E9 05 byte 09](./e9-05.md#byte-09---mask--color) (mask+color split,
same 32-entry palette) until proven otherwise.

## Building Your Own

```text
83 01 <E1|E2> 00 E9 04 00 <TIMING> 0E <COLOR?>
```

Structure is a best-effort extrapolation from E9 05 minus the vibration byte — **not**
independently confirmed end-to-end. Verify against a live capture before relying on it for a
show.

## Suggested Lab Tests

1. **Byte offset confirmation** — capture several E9 04 packets with known, distinct colors
   selected in-park (if possible) and determine definitively whether the color value lives at
   byte 7, byte 9, or elsewhere. This is the single most important open question for this
   opcode.
2. **Timing model confirmation** — once the color byte position is settled, run the same
   on-time/fade timing sweep already done for E9 05 to confirm whether E9 04 shares that exact
   model or has its own constants (as E9 0E turned out to, despite initially looking like a
   simple E9 05 variant).
3. **Byte 08 sweep** — confirm whether `0x0E` is a required constant (like E9 0C/E9 0D/E9 0E's
   `0x0F` marker) or coincidental across the small sample set collected so far.
