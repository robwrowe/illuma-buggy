# E90E Packets - Five-Zone Color (Strobe)

Lights the five LED zones (Center, TopRight, BottomRight, BottomLeft, TopLeft) each in an
independently-assigned color, animated as a strobe. The specific strobe *pattern* (tight-sync
flash, directional edge, or chaotic) is selected by byte 13 rather than by a simple on/off
flag — see below.

```text                                                         Byte 13

0x8301 E200 E90E 00 82 0F BB BB BB BB BB 59 32 02 AE ────────── 0x32 ── directional edge-in
0x8301 E100 E90E 00 05 0F BB B5 A4 A4 B5 59 19 02 AE ────────── other ─ chaotic / base strobe
```

Byte numbering above includes the `8301 <E1|E2> 00` prefix, matching [E9 05](./e9-05.md) and
[E9 0C](./e9-0c.md). Raw lab captures typically omit it.

**Byte offset note:** the pattern-shape selector lives at byte offset 13 (not 15) in
full-frame indexing — this offset has previously been a source of rule-matching bugs; double
check hex-vs-decimal when configuring a `byte`-type match condition against it (`0x19` is 25
decimal, `0x32` is 50 decimal, `0x50` is 80 decimal — entering `19`, `32`, or `50` directly as
decimal instead of converting from hex silently matches the wrong byte value rather than
failing loudly).

## Byte Details

### Byte 07 - Timing

Same bit layout as [timing byte detail](../timing-byte.md), with an **E9 0E-specific timing
model** (`tm_e90e` in the current rules engine), distinct from the `tm_default` model used by
E9 05/E9 09:

- **On-time, scaler=0**: `1.5×t` (note: `1.5`, not E9 05/E9 09's `1.6` — confirmed as a real,
  separate constant for this opcode, not a typo carried over).
- **On-time, scaler=1**: `3.0×t`, same constant as the default model.
- **Extended-timeout mode (bit 7)**: `7.6×t`, same constant as the default model.
- **`t=0` fallback**: `3` seconds, same as the default model.
- **Fade-out step**: `0.25s` per fade_bits step (vs the default model's `0.5s` step) — i.e.
  fade_bits `00/01/10/11` → `0/0.25/0.5/0` seconds. **Note the 4th value is `0`, not `0.75`** —
  the fade-stretch table for this model is `[0, 0.5, 1, 0]` in the shipped timing model config,
  which does not follow a simple linear `fade_bits × 0.25` pattern at the high end. Treat
  fade_bits=`11` as a confirmed-zero-stretch case, not a gap in testing.
- **fade_bits=`11` special case**: separately documented as producing a fixed ~1.5s slow-flash
  behavior in earlier lab notes — consistent with fade_bits=`11` mapping to `0` *stretch* in
  the table above while the strobe pattern itself (not the fade) accounts for the visible
  ~1.5s cadence. These are two different mechanisms (fade-stretch vs strobe-pattern timing);
  don't conflate them when reading the timing model.

### Byte 08 - Length/Marker

Constant `0x0F` across observed samples, same role as E9 0C.

### Bytes 09-13 - Per-Zone Color

Same TopLeft → BottomLeft → BottomRight → TopRight → Center order and mask+color split as
[E9 0C](./e9-0c.md#bytes-09-13---per-zone-color) — confirmed via the shipped rules engine
implementation, which extracts each zone at `bitCount: 5` (color-index only, consistent with
the mask=`0b101`-fixed anomaly documented for E9 0C).

### Byte 13 - Pattern-Shape Selector

Confirmed via four independently-tuned rules currently shipped in the rules engine, each
matching a distinct value (or value set) at this byte and driving a different WLED effect/
speed combination:

| Byte 13 value | Pattern | WLED effect | Speed (`sx`) |
| -------------- | ------------------------- | ----------- | ------------- |
| `0x19` (25) or `0x50` (80) | Tight-sync flash | 74 | 100 |
| `0x32` (50) | Directional edge-in | 67 | 20 |
| bit[6] or bit[7] of byte 5 set (see note) | Slow/extended strobe | 74 | 20 |
| anything else (no match above) | Chaotic / base strobe (catch-all) | 67 | 20 |

The "slow" rule matches on byte-5 timing bits (extended-timeout or scaler flag) rather than
byte 13 directly — i.e. it's really keying off the *timing byte's* extended/scaler bits, not
an independent pattern-shape value, and is checked before the byte-13-based rules in priority
order. This suggests the "slow flash" behavior your earlier notes associated with fade_bits=
`11`/byte-13 `0x19` may actually be driven by the timing byte's own extended-mode bit rather
than byte 13 at all — worth re-examining whether byte 13's `0x19` value is doing independent
work or is usually just co-occurring with an extended-timeout timing byte in your captured
samples.

The catch-all "chaotic/base strobe" rule has no byte-13 condition at all — it matches any
E9 0E packet that didn't already match one of the three more specific rules above it in
priority order, and uses fan-out extraction (each zone's color applied to 4 physical LED
segments instead of 1) rather than the single-segment-per-zone mapping the other three
variants use.

Only three specific values (`0x19`, `0x32`, `0x50`) have dedicated rules; all other values
fall through to the catch-all. This is consistent with the opcode doc's original working
theory that byte 13 is a small enumerated selector rather than a bitfield, but the full set of
valid values is still not exhaustively confirmed — only these three plus "anything else" have
been distinguished in practice.

### Byte 18 - Required Constant

`0xAE` across observed samples — confirmed required for this opcode (matches your project's
existing E9 0E findings). Not yet tested whether other values are silently accepted with
different behavior or rejected outright.

### Bytes 15-17 - Tail

Byte 16 appears constant (`0x59`) across samples; bytes 15 and 17 vary. Previously documented
theory: byte 16 bit-5 controls flash count (6 vs 4 flashes) — not independently re-confirmed
against the current rule set, which doesn't extract or condition on these bytes at all. Treat
as unconfirmed pending a dedicated sweep.

## Building Your Own

```text
83 01 <E1|E2> 00 E9 0E 00 <TIMING> 0F <ZONE_TL> <ZONE_BL> <ZONE_BR> <ZONE_TR> <ZONE_C> <PATTERN> <TAIL x2> AE
```

### `<TIMING>` (byte 07)

Use the E9 0E-specific model above, **not** the E9 05/E9 09 default model — the `1.5×t`
scaler-0 constant and `0.25s` fade step are both different from those opcodes.

### `<ZONE_*>` (bytes 09-13)

Same construction as [E9 0C](./e9-0c.md#zone_-bytes-09-13) —
`zone_byte = (0b101 << 5) | color_index`.

### `<PATTERN>` (byte 13)

`0x19` or `0x50` for tight-sync flash, `0x32` for directional edge-in, any other value for the
default/chaotic strobe pattern.

### Byte 18

Always `0xAE`.

## Suggested Lab Tests

1. **Byte 13 exhaustive sweep** — only 3 distinct values have confirmed behavior; a full
   0x00-0xFF sweep (or at least a representative sample across the range) would confirm
   whether this is truly a small enum or has more valid states.
2. **Byte-5 vs byte-13 interaction isolation** — construct packets that independently vary the
   timing byte's extended/scaler bits and byte 13's pattern value in all four combinations, to
   determine whether "slow flash" is driven by byte 5, byte 13, or genuinely requires both.
3. **Tail byte sweep (bytes 15, 17)** — hold everything else constant, vary one byte at a time,
   watching specifically for flash-count changes per the byte-16-bit-5 theory.
