# E90D Packets - Five-Zone Color (Strobe Family)

Lights the five LED zones (Center, TopRight, BottomRight, BottomLeft, TopLeft) each in an
independently-assigned color from the internal palette. Structurally identical to
[E9 0C](./e9-0c.md) and [E9 09](./e9-09.md)'s zone-color layout, and shares E9 0E's strobe
timing model in the current rule implementation — see caveat under Byte 07 below.

```text
0x8301 E100 E90D 00 1F 0F B9 B9 B9 B9 B9 59 19 02 48 AE ────── all green, strobe pattern, ~24s
0x8301 E100 E90D 00 13 0F B4 B5 B5 B5 B3 59 0F FB 48 AE ────── 4 red / 1 yellow, ~5.5s
  ││││ ││││ ││││ ││ ││ ││ ││ ││ ││ ││ ││ ││ ││ ││ ││ ││
  ││││ ││││ ││││ ││ ││ ││ ││ ││ ││ ││ ││ ││ ││ ││ └┴──────── Byte   18 - 0x48/0xAE tail (unconfirmed, see below)
  ││││ ││││ ││││ ││ ││ ││ ││ ││ ││ ││ ││ ││ ││ └┴─────────── Byte   17 - Tail (unconfirmed)
  ││││ ││││ ││││ ││ ││ ││ ││ ││ ││ ││ ││ ││ └┴────────────── Byte   16 - Tail (unconfirmed)
  ││││ ││││ ││││ ││ ││ ││ ││ ││ ││ ││ ││ └┴───────────────── Byte   15 - Tail (unconfirmed)
  ││││ ││││ ││││ ││ ││ ││ ││ ││ ││ ││ └┴──────────────────── Byte   14 - 0x59 constant (unconfirmed)
  ││││ ││││ ││││ ││ ││ ││ ││ ││ ││ └┴─────────────────────── Byte   13 - Zone 5 (Center)
  ││││ ││││ ││││ ││ ││ ││ ││ ││ └┴──────────────────────────Byte   12 - Zone 4 (TopRight)
  ││││ ││││ ││││ ││ ││ ││ ││ └┴────────────────────────────  Byte   11 - Zone 3 (BottomRight)

  └┴┴┴─────────────────────────────────────────────────────  Bytes 0-1 - Disney Identifier
```

Note: raw lab captures for this opcode typically start at the OP code (`E9 0D ...`) without the
`8301 <E1|E2> 00` prefix, same as E9 0C/E9 0E. Byte numbering above includes that prefix for
consistency with the rest of these docs.

**A second, distinct payload shape exists for this opcode and is not yet covered by this
doc or the current rules-engine implementation:**

```text
0x8301 E200 E90D 00 64 D2 55 FC 00 00 D2 58 64 42 00 78 ── Yellow Outer, Red Inner
0x8301 E100 E90D 00 67 D2 55 2B 2E 00 D2 58 24 00 FC 78 ── Purple Outer, Yellow Inner
```

This branch is marked by a `D2 55` byte pair (instead of the `0F` zone marker) at byte 08-09,
followed by what decodes as a raw RGB triple, then a second `D2 58` marker and a second RGB
triple further in the payload — i.e. two independent inner/outer color blocks, not five
palette-indexed zones. See the `e9-07_solid_rgb` rule's `channelGroup` extract shape for the
closest existing implementation pattern (raw RGB, not palette), though a dedicated dual-block
E9 0D rule hasn't been built yet. **The currently-shipped `e9-0d_five-color_base_strobe` rule
only matches and handles the zone-color branch shown in the main diagram above — packets using
the `D2 55`/`D2 58` shape will not match it and will fall through to lower-priority rules.**

## Byte Details

### Byte 07 - Timing

Same bit layout as [timing byte detail](../timing-byte.md). The current rules-engine
implementation (`e9-0d_five-color_base_strobe`) applies the `tm_e90e` timing model (see
[E9 0E](./e90e.md#byte-07---timing)) to this opcode rather than a dedicated E9 0D model —
this is a working assumption carried over from E9 0E's structural similarity, not something
independently confirmed against E9 0D-specific timing sweeps. Treat E9 0D on-time/fade
predictions as unverified until tested directly.

### Byte 08 - Length/Marker

`0x0F` in the zone-color branch (see diagram) — same role as E9 0C/E9 0E's byte 08. The RGB
branch uses `D2 55` at this position instead (see above); the two branches are mutually
exclusive ways of encoding this opcode's payload, not two fields.

### Bytes 09-13 - Per-Zone Color

The shipped rule (`e9-0d_five-color_base_strobe`) extracts 5 bytes at offsets 7-11
(`bitCount: 5` each, color-index only) targeting the same segment IDs used by
[E9 09's rule](./e909.md) (`segnysug0j`, `segqq2wflj`, `segqq2w24v`, `segqq2wn7j`,
`segqq2wwrt`, in that order) — **this means the shipped rule actually follows E9 09's
Center-first clockwise order (C, TR, BR, BL, TL), not the TL-first order** this doc's diagram
above shows. The diagram's TL-first labeling was carried over from E9 0C's documented order by
structural analogy and has not been independently re-confirmed for E9 0D against the order the
shipped rule actually implements. Same open question flagged in the [E9 08 doc](./e908.md#zone-branch---per-zone-color)
for its zone branch — worth resolving with a one-zone-at-a-time visual test before trusting
either convention for E9 0D specifically. Until then, treat the shipped rule's implementation
(E9 09-style Center-first order) as the operative behavior, and this doc's diagram labels as
the unconfirmed legacy assumption.

Same mask+color split and mask=`0b101` fixed-value anomaly as
[E9 0C](./e9-0c.md#bytes-09-13---per-zone-color) otherwise — not independently re-confirmed
for E9 0D specifically, carried over based on identical byte-width structure in the shipped
rule.

### Bytes 14-18 - Tail

Purpose unknown, same as [E9 0C's tail bytes](./e9-0c.md#bytes-15-17---tail). Byte 18 in the
two samples above (`AE` in both) matches E9 0E's confirmed byte-18 `0xAE` strobe-mode
constant — worth testing whether E9 0D shares that same required-constant behavior, since the
current rule already borrows E9 0E's timing model.

## Building Your Own

### Zone-color branch (currently implemented)

```text
83 01 <E1|E2> 00 E9 0D 00 <TIMING> 0F <ZONE_TL> <ZONE_BL> <ZONE_BR> <ZONE_TR> <ZONE_C> <TAIL x5>
```

Same construction as [E9 0C](./e9-0c.md#building-your-own)'s zone-color formula — `zone_byte =
(0b101 << 5) | color_index`.

### RGB branch (not yet implemented in rules engine)

```text
83 01 <E1|E2> 00 E9 0D 00 <TIMING> D2 55 <R1> <G1> <B1> D2 58 <R2> <G2> <B2> <TAIL>
```

Unconfirmed whether `D2 55`'s block always represents "outer" and `D2 58` always "inner," or
whether that mapping is sample-dependent — the two lab samples above didn't cleanly confirm
consistent block-to-role assignment on first inspection. Needs an isolation test (send one
block as pure red, the other as off, and see which zone lights) before building a rule against
it — see the E9 0D entry in the rules-engine test-packet notes for a specific test proposal.

## Suggested Lab Tests

1. **E9 0D-specific timing sweep** — the current rule inherits E9 0E's timing model
   (`tm_e90e`) untested against E9 0D directly. Hold zone colors constant, sweep byte 7 across
   representative `t`/scaler/fade combinations, and compare observed on-time/fade against
   `tm_e90e`'s predictions to confirm or refute the shared-model assumption.
2. **RGB-branch block isolation** — send `D2 55 FF 00 00 D2 58 00 00 00` (block 1 = pure red,
   block 2 = off) and the inverse, to determine which block controls which physical
   inner/outer zone.
3. **Byte 18 constant check** — test whether `0xAE` is required for the zone-color branch the
   way it is for E9 0E, or whether E9 0D tolerates other values there.
