# E908 Packets - Single Color (RGB) or Five-Zone Color (Palette)

Two structurally distinct payload shapes share this opcode, selected by a marker at byte 08 —
the same branch-selection pattern used by [E9 07](./e907.md). One branch is a single raw RGB
color across all five LEDs; the other is a five-zone palette-indexed color, structurally
identical to [E9 09](./e909.md)'s zone layout. Both branches are currently implemented as
separate shipped rules.

```text                                                    RGB (R,G,B) / Zones (C,TR,BR,BL,TL)
                                                            ─────────────────────────────────
0x8301 E100 E908 00 21 D2 55 45 64 61 BA ────────────────  69, 100, 97 ── "solid white"
0x8301 E100 E908 00 4F 0F AF A4 B5 B5 A4 ────────────────  15/04/21/21/04 ── UL+UR blue, BL+BR red, C yellow
  ││││ ││││ ││││ ││ ││ ││ ││ ││ ││ ││ ││
  ││││ ││││ ││││ ││ ││ ││ ││ ││ ││ ││ └┴─ Byte   11 - RGB branch: n/a / Zone branch: Zone 5 (Center)
  ││││ ││││ ││││ ││ ││ ││ ││ ││ ││ └┴──── Byte   10 - Blue (RGB) / Zone 4 - TopRight (zone branch)
  ││││ ││││ ││││ ││ ││ ││ ││ ││ └┴─────── Byte   09 - Green (RGB) / Zone 3 - BottomRight (zone branch)
  ││││ ││││ ││││ ││ ││ ││ ││ └┴────────── Byte   08 - Red (RGB) / Zone 2 - BottomLeft (zone branch, offset shifts — see note)
  ││││ ││││ ││││ ││ ││ ││ └┴───────────── Byte   07 - Timing
  ││││ ││││ ││││ ││
  ││││ ││││ ││││ └┴────────────────────── Byte   06 - 0xD2 0x55 marker (RGB) / 0x0F marker (zone) — see note
  ││││ ││││ ││││
  ││││ ││││ └┴┴┴───────────────────────── Bytes 4-5 - 0xE908 - OP Code for Effect
  ││││ └┴┴┴────────────────────────────── Bytes 2-3 - Purpose Unknown
  └┴┴┴──────────────────────────────────── Bytes 0-1 - 0x8301 - Disney Identifier
```

**Byte-offset note:** unlike E9 07 (where the branch marker sits at byte 08, after a
conventional byte-07 timing byte), the shipped rules for E9 08 place the branch marker at
**byte 06** (`offset: 6` in both rules' match conditions) and the timing byte at byte 07 comes
*before* it in read order but the marker check happens on byte 6, not 8. Concretely: the
zone-color extract offsets in the shipped rule start at 7 (`bitCount: 5` each, offsets 7-11),
meaning the marker byte the rule actually keys off (`offset: 6`) is one byte earlier than
where E9 07's marker sits. Treat the diagram's byte numbers as relative to the shipped rule's
own offsets, not as independently re-derived — this opcode's exact full-frame byte alignment
relative to the other E9 0x opcodes hasn't been independently cross-checked end-to-end.

## Byte Details

### Bytes 0-1, 2-3, 4-5, Spacer

Same as every other E9xx opcode — Disney identifier, unknown (E100/E200), op code, spacer.

### Byte 06 - Branch Marker

Confirmed via two shipped rules (`e9-08_solid_rgb`, `e9-08_five-color_solid`), both matching
on a `byte` condition at offset 6:

- `0xD2` (210 decimal) — selects the **RGB branch**. Implemented by `e9-08_solid_rgb`.
- `0x0F` (15 decimal) — selects the **five-zone palette branch**. Implemented by
  `e9-08_five-color_solid`.

Same two-branch pattern as E9 07, though the marker here is a single byte (`0xD2`) rather than
E9 07's 2-byte `D2 55` sequence — worth double-checking whether E9 08's RGB branch also has a
second marker byte immediately after (mirroring E9 07's `55`) that just isn't being checked by
the current match condition, or whether the two opcodes' branch markers are genuinely
different lengths. Not resolved either way; flagged as an open question.

### RGB Branch - Color

Implemented as a `channelGroup` extract at offsets 8/9/10 (R/G/B), currently using
`scale: "bitReplicate6to8"` with `bitStart: 1, bitCount: 6` — **the same 6-bit-packed scheme
flagged as likely-incorrect for E9 07's RGB branch.** The one available labeled sample for
this branch (`45 64 61` → "solid white") is close to a neutral gray/white blend under a direct
8-bit reading (R=69, G=100, B=97 — roughly balanced across channels, consistent with
"white"-ish), which is at least plausible evidence for `direct8` over `bitReplicate6to8` here
too, though with only one sample this isn't as strongly confirmed as E9 07's 5-sample RGB
table. Recommend applying the same `direct8` scale fix here as noted for E9 07, and validating
against a few more labeled RGB samples before fully trusting either scale mode for this
branch.

Separately, several `D2 55 00 00 00`-shaped E9 08 samples were observed going to solid black —
consistent with `(0,0,0)` being a legitimate valid "off" RGB value under the direct-RGB theory,
not a decode failure (see E9 07 doc for the fuller discussion of this same pattern).

### Zone Branch - Per-Zone Color

Implemented via 5 separate extract entries (`center-color`, `top-right-color`,
`bot-right-color`, `bot-left-color`, `top-left-color`), each `bitCount: 5` (color-index only,
consistent with the mask=fixed-value pattern seen throughout this protocol family), at offsets
7 through 11 — Center first, then clockwise TR → BR → BL → TL, matching
[E9 09's confirmed zone order](./e909.md#bytes-9-13---mask--color) exactly (not the E9 0C/E9
0D/E9 0E family's TL-first order — worth double-checking which convention actually applies
here if you see zone-position mismatches in testing, since E9 08's zone branch structurally
resembles the E9 0C/0D/0E family in most other ways but uses E9 09's zone-order convention per
the shipped rule).

## Building Your Own

### RGB branch

```text
83 01 <E1|E2> 00 E9 08 00 <TIMING> D2 <R> <G> <B>
```

Unconfirmed whether a second marker byte (`55`, matching E9 07) belongs after `D2` — copy from
a working captured sample until this is resolved.

### Zone branch

```text
83 01 <E1|E2> 00 E9 08 00 <TIMING> 0F <ZONE_C> <ZONE_TR> <ZONE_BR> <ZONE_BL> <ZONE_TL>
```

`zone_byte = color_index` (5-bit, per the shipped rule's `bitCount: 5` extraction — unclear
whether a mask nibble also exists above these 5 bits the way it does in the E9 0C/0D/0E
family, since the rule only extracts 5 bits and doesn't condition on the rest of the byte).

## Suggested Lab Tests

1. **Confirm/deny the second RGB marker byte** — determine whether E9 08's RGB branch marker
   is a single `0xD2` byte or a 2-byte `D2 55` sequence like E9 07's.
2. **RGB scale mode** — same fix/validation needed as E9 07: gather more labeled RGB samples
   and confirm `direct8` over `bitReplicate6to8`.
3. **Zone-order cross-check** — confirm whether the zone branch truly follows E9 09's
   Center-first clockwise order (as currently implemented) or E9 0C's TopLeft-first order, with
   a test that lights one zone at a time and visually confirms position.
4. **Mask-bit check** — confirm whether zone bytes in this opcode carry a mask nibble above the
   5-bit color index (as the E9 0C/0D/0E family's bytes structurally do, even if that mask
   value is fixed/ignored) or whether E9 08's zone bytes are genuinely just a bare 5-bit value
   with nothing above them.
