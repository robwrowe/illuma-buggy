# E913 Packets - Base + Chase / Paired-Alternation Color (Not Yet Implemented)

**No rule currently exists for this opcode in the rules engine**, despite this being the
**best-understood opcode in the E9 10-E9 14 family** — most of the confirmed structural
findings for the whole family (base+chase pattern, paired-alternation, timing-byte
confirmation) came from controlled byte-isolation testing run specifically against this
opcode. This doc reflects that stronger evidence base; the sibling docs for
[E9 10](./e910.md), [E9 11](./e911.md), [E9 12](./e912.md), and [E9 14](./e914.md) carry these
findings over as working assumptions, not independent confirmations.

## Confirmed Timing Behavior

Directly confirmed via a timing-byte sweep on this opcode specifically — hold zone colors
constant, vary only the timing byte:

| Timing byte | Total on-time | Cycle time |
| ----------- | -------------- | ----------- |
| `0x05` | ~8s | ~0.5s |
| `0x0F` | ~24s | ~0.5s |
| `0x30` | ~1.5s | ~0.5s |
| `0xB0` | ~7.5s | ~1s |

Matches [timing-byte.md](../timing-byte.md)'s bit layout directly: `0xB0` and `0x30` both have
bit 6 (scaler) set and land at the longer ~1s cycle, while `0x05`/`0x0F` (scaler unset) land at
~0.5s — bits [3:0] (time value) track total on-time independent of cycle speed, as the model
predicts. This is the strongest, most directly-confirmed timing result anywhere in the E9
10-E9 14 family — safe to build a rule's timing block against with real confidence, unlike the
color-byte theories below.

## Confirmed Color-Byte Structure — Two Distinct Sub-Patterns

**These are the two patterns referenced throughout the sibling E9 10/11/12/14 docs — this is
where they were actually isolated and confirmed.**

### Pattern A — "Base + chase"

Confirmed via direct byte-isolation testing (holding all other bytes constant, changing one
zone byte at a time):

```
5D 44 58 F4... → dim blue (base),    vivid ??? chase
5D 4E 58 F4... → dim blue (base),    vivid RED chase
4E 44 58 F4... → dim PURPLE (base),  vivid ??? chase (pink chase)
5D 59 58 F4... → dim blue (base),    vivid GREEN chase
59 44 58 F4... → dim GREEN (base),   teal chase
```

**Zone-byte 0 = "dim/base" color. Zone-byte 1 = "vivid/chase" color.** Independently
controllable — overriding byte 0 alone changes only the base color; overriding byte 1 alone
changes only the moving/vivid highlight. This is a two-role system (one static base color, one
independently-animated highlight), not five (or four) independent positional zones.

### Pattern B — "Paired-alternation"

Confirmed via a separate set of samples, all sharing zone-byte-0 = `0xBD`:

```
BD A4 A4 A4 A4 → outer royal blue, alternating pairs
BD B5 AF B5 AF → red/yellow alternating OUTER, inner OFF
BD B5 BC B5 BC → alternating red/white on OUTER
BD A0 B6 A0 B6 → alternating cyan/teal & white, INNER dark
```

In every sample above, **zone-byte 1 == zone-byte 3, and zone-byte 2 == zone-byte 4** — i.e.
the 4 bytes following byte 0 aren't 4 independent values, they're two alternating pairs
(`[pairA, pairB, pairA, pairB]`) sharing state, while byte 0 governs a separate base/inner
role. Confirmed across 4 independent samples with perfect consistency.

**One sample breaks this pattern** (`A2 A8 B1 B9 A2`, described as "alternating 2 at a time,
center never on") — its byte 0 (`A2`) doesn't match the `BD` constant seen in every Pattern-B
sample above, suggesting this may be a third, distinct sub-format rather than a Pattern-B
variant. Not resolved — flagged as an open question rather than forced into either pattern.

### Which pattern applies when?

Not yet determined from a single discriminating byte — Pattern A and Pattern B samples don't
share an obviously-different marker value at any consistent position checked so far. This is
the single most important open question for turning this opcode's theory into a working rule:
without knowing which byte (if any) selects between the two patterns, a rule can only
implement one pattern's behavior and will visually misrender packets using the other.

## Building Your Own

Not yet buildable as a single general-purpose rule given the unresolved Pattern A/B selector
question — but a rule targeting *one* pattern specifically (e.g. Pattern A only, matching
whatever byte signature turns out to correlate with it) is more achievable than for any other
opcode in this family, given the confirmed color-byte structure and timing model above.

## Suggested Lab Tests

1. **Pattern A/B selector isolation** — this is the priority. Gather more samples from both
   patterns and systematically compare every byte position (not just the zone-color bytes) to
   find a consistent discriminator. Check the trailer bytes specifically, since neither pattern
   write-up above has ruled out a trailer-based selector.
2. **`A2 A8 B1 B9 A2` anomaly follow-up** — determine whether this is a third pattern or an
   edge case of Pattern B with an atypical byte-0 value.
3. **Trailer investigation** — same recurring-chunk pattern seen in E9 11/E9 12/E9 14
   (`D1 46`, `D0 65`, etc.) appears here too; worth checking whether it correlates with
   anything now that the color-byte structure is well understood, freeing up analysis capacity
   to focus on the trailer specifically.
