# E90F Packets - Inner/Outer Color Pair (Not Yet Implemented)

**No rule currently exists for this opcode in the rules engine.** Packets matching E9 0F will
fall through to the unhandled-opcode fallback (see the
[rules engine reference](./rules-engine-reference.md#known-issue-in-the-current-ruleset) for a
caveat about that fallback's current match condition). This doc captures the working theory
from lab observation, for use when building a rule.

Unlike the five-independent-zone family (E9 09/E9 0C/E9 0D/E9 0E), E9 0F's lab notes
consistently describe behavior in terms of **inner** and **outer** color roles rather than
five discrete corner positions — every collected sample's observation notes explicitly use
"inner"/"outer" language, and none describe distinct per-corner colors the way E9 09 samples
do.

Byte numbers are intentionally left blank in the diagram above — this opcode has not been
built into a rule yet, so no extract offsets have been validated the way E9 07/E9 08/E9 0D's
diagrams (which follow their shipped rules' own offset conventions) have been. Treat this
purely as a payload-shape description pending implementation.

**A second payload family exists under this opcode** with a different trailer signature
(`D1 41`/`D1 46` instead of `D2 46`, sample third row above) — not enough samples exist to
determine whether this is a genuinely different sub-mode or the same format with different
trailer content. Flagged for awareness, not yet workable.

## Observed Behavior

Every sample collected describes one of: ping-pong pulsing between inner and outer, fast
shimmer on one role while the other is off, or a color-loop cycling all five LEDs through one
shared color (one sample only — possibly a distinct third sub-mode, or the "inner"/"outer"
distinction collapsing to a single value in that specific packet). No sample describes five
independently-colored corners.

**Byte-role ambiguity, not yet resolved:** which of the two candidate color bytes maps to
"inner" vs. "outer" has not been isolated with a controlled test. Two samples above show the
same two colors (white, blue) in opposite byte positions with opposite inner/outer labels in
the notes — consistent with a straightforward two-byte-swap, but this hasn't been confirmed
with a test that holds one byte fixed and only varies the other.

## Building Your Own

Not yet buildable with confidence — the byte-role ambiguity above needs resolving first. If
experimenting, the `0F` marker byte position and general structure (timing byte, then `0F`,
then two color-ish bytes, then a longer trailer) can be copied from a working sample and
mutated one byte at a time.

## Suggested Lab Tests

1. **Inner/outer byte isolation** — send a baseline sample, then override only the first
   candidate color byte to a distinct, easily-identified color (e.g. deep blue, palette index
   4) while leaving everything else constant. Repeat for the second candidate byte. Whichever
   byte's change visibly affects the "inner" ring vs. "outer" ring settles the role assignment.
2. **Timing byte sweep** — no dedicated sweep has been run for this opcode; confirm whether it
   follows the standard timing model or has its own constants (E9 0E turned out to need its
   own despite initial appearances).
3. **Second payload family investigation** — gather more samples of the `D1 41`/`D1 46`
   trailer variant to determine if it's a distinct sub-mode worth its own rule.
4. **Trailer chunk investigation** — the recurring `48 8D` / `D2 46` chunks appear across
   multiple samples regardless of visual behavior, suggesting they may be fixed animation-mode
   identifiers rather than per-call parameters (the same pattern already confirmed for parts of
   the E9 10-E9 14 family — see those docs). Worth checking whether E9 0F's trailer is actually
   a small enumerated animation-mode selector once more samples exist.
