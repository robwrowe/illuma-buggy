# E914 Packets - Base + Chase Color, 5-Zone + Extra Param (Not Yet Implemented)

**No rule currently exists for this opcode in the rules engine.** This doc captures the
working theory from lab observation, for use when building a rule.

Largest member of the [E9 10](./e910.md)-[E9 13](./e913.md) family — 5 zone-color bytes (same
count as E9 13) plus one additional parameter byte not present in the shorter family members.
A second, structurally distinct payload branch also exists — see below.

Byte numbers left blank — no shipped rule exists yet. The second sample above (`D0 37 F0 D2
3D...`) shows a **completely different opening structure** from the first — no `0F` marker,
no obvious 5-byte zone-color block in the same position. This may be the same `d2 5x`-family
raw-RGB branch confirmed for E9 07/E9 08/E9 0D (see [E9 0D](./e90d.md) for that family's
write-up), or a third, independent format — **not confirmed either way**. Don't assume the
diagram above (based on the `0F`-marker branch) applies to this second sample shape.

## Observed Behavior

Samples split roughly into two groups:

1. **"Circle animation"** samples (the more common group in the collected data) — teal/blue,
   consistent ~16-26s on-time, described similarly to E9 12's circle-chase behavior.
2. **"Slow fade in/out"** samples — all five LEDs sharing one color, ~19s total with ~1.5s
   cycle, no positional chase description. These samples also structurally match the
   non-`0F`-marker payload shape shown as the second example above, so this behavioral split
   may directly correspond to the two payload-shape branches rather than being two sub-modes
   of one shape.

## Color-Byte Structure (working assumption, unconfirmed for this opcode specifically)

Carried over from [E9 13](./e913.md#confirmed-color-byte-structure--two-distinct-sub-patterns),
which has the strongest direct evidence in this family — E9 14's 5 zone-color bytes are
assumed to follow either the "base + chase" or "paired-alternation" pattern documented there,
but **this has not been independently isolated for E9 14**. Given E9 14 is the one member of
the family with an extra trailing parameter byte beyond the 5 zone-color bytes, it's also
possible that extra byte selects between color-byte sub-patterns (unlike E9 13, which doesn't
have an equivalent byte) — worth checking specifically, since this opcode is structurally
positioned to answer the "what selects Pattern A vs Pattern B" question left open in the E9 13
doc.

## Timing

Standard family timing model assumed (see [E9 13](./e913.md#confirmed-timing-behavior) for the
directly-confirmed version) — not independently tested for E9 14.

## Building Your Own

Not yet buildable with confidence — resolve the two-payload-shape question first (confirm
whether the non-`0F` samples are the `d2 5x` RGB family or something else), then apply the
same approach as [E9 13](./e913.md#building-your-own) for the `0F`-marker branch once the
color-byte pattern selector is understood.

## Suggested Lab Tests

1. **Two-payload-shape resolution** — determine whether the non-`0F`-marker samples belong to
   the `d2 55`/`d2 58` RGB family (test with the same block-isolation approach used for
   [E9 0D](./e90d.md#suggested-lab-tests)) or are a genuinely separate format.
2. **Extra-parameter-byte investigation** — this is the one byte E9 14 has that no shorter
   family member does; test whether it selects between the base+chase and paired-alternation
   color patterns, which would resolve an open question for the whole family, not just E9 14.
3. **Timing confirmation** — run the same sweep methodology already used on E9 13.
4. **Circle-animation vs. fade-in/out correlation** — confirm whether the two observed
   behavior groups map cleanly onto the two payload shapes, or whether both behaviors can occur
   within the `0F`-marker branch alone.
