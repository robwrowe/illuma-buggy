## Byte Details

### Bytes 0-1 - Disney Identifier

Always the Disney identifier. Required.

### Bytes 2-3 - Unknown

Purpose unknown. Can be omitted.
Other values: `0xE100`

### Bytes 4-5 - Op Code

OP Code for effect. Required.

### Byte 06 - Spacer

Spacer. Required.

### Byte 07 - Timing

Timing formula. See [timing byte detail](../timing-byte.md) for details.

**Confirmed against the 12 packets above** — the E9 05 timing model carries over to E9 09
unchanged, with one correction and one major new finding from this batch:

- **On-time, scaler=0**: `1.6×t` confirmed again across 4 more packets here (`0x01`, `0x11`,
  `0x23`, `0x13`), all within the same small residual (~0.1-0.3s) seen previously.
- **On-time, scaler=1**: `3.0×t` confirmed again (`0x65`: t=5 → predicted 15.0s, observed
  15s — exact).
- **Extended-timeout mode (bit 7), scaler=0**: `7.6×t` confirmed again (`0x82`: t=2 →
  predicted 15.2s, observed 15s — extends this formula's confirmation to a 6th data point).
- **fade_bits=`10` correction**: previously listed as "untested, but fits the ×0.5s pattern"
  at a predicted 1.0s. Now tested (`0x65`, `0x2c`, `0x23`) and the results don't agree with
  each other: `0x2c` and `0x23` both show 1.0s (matching the ×0.5s pattern), but `0x65` shows
  0.5s instead — identical to what fade_bits=`01` produces. This is a real, unresolved
  inconsistency, not yet explained. It's possible fade_bits=`10`'s actual fade time depends on
  something else in the packet (scaler bit? `t` value?) rather than being a fixed 1.0s — worth
  a dedicated isolated test before trusting this value either way.
- **New finding — bit7=1 combined with scaler=1 is its own case, not covered by the
  extended-timeout formula above.** `0xFE` (`t=14`, always=1, scaler=1) was observed at
  **210 seconds** — matching a clean `15 × t` formula exactly (`15×14=210`), a different
  constant from anything else confirmed so far. Only one data point, so treat this formula as
  a strong lead rather than fully proven.
- **`t=15` (the maximum 4-bit value, `0b1111`) looks like a second reserved sentinel**, the
  same way `t=0` was found to be one. `0xFF` (`t=15`, always=1, scaler=1) is labeled "Always
  On" rather than a specific duration — if the `15×t` formula above applied literally, `t=15`
  would predict 225s, not indefinite persistence. The more likely explanation, consistent with
  how `t=0` behaves, is that `t=15` under this specific always+scaler combination is a
  dedicated "hold indefinitely until a new packet arrives" sentinel, separate from the
  computed formula used at `t=0`-`14`. This would also resolve an earlier open question: a
  previous isolated test of the "always-on" bit alone (without `t=15`) did not stay on
  indefinitely — which makes sense under this theory, since true indefinite hold may require
  the specific `t=15` sentinel, not just the bit being set.

### Byte 08 - Unknown.

Purpose unknown. Required.

### Bytes 9-13 - Mask & Color

Discrete colors starting from the center, then going clock-wise from the top right LED.

See [Color and Mask Palette](../color-and-mask-palette.md) for details.

**Confirmed zone order:** Center, Top-Right, Bottom-Right, Bottom-Left, Top-Left, going
clockwise from top-right. This overrides an earlier finding from a different round of lab
testing that had concluded Center, Top-Left, Bottom-Left, Bottom-Right, Top-Right — that
earlier order was derived from packets that turned out not to be asymmetric enough to
distinguish the two orderings reliably. If any other doc or note still references the older
[C, TL, BL, BR, TR] order, treat this confirmed [C, TR, BR, BL, TL] order as authoritative
and update accordingly.

### Byte 14 - Vibration Pattern

See [EMCOT.world](https://emcot.world/Disney_MagicBand%2B_Bluetooth_Codes#Vibration_Palette) for more information.

## Building Your Own

```text
83 01 <E1|E2> 00 E9 09 00 <TIMING> 0F <MASK+COLOR> <MASK+COLOR> <MASK+COLOR> <MASK+COLOR> <MASK+COLOR> <VIBRATION>
```

### `<TIMING>` (byte 07)

Same confirmed model as E9 05 — see [timing byte detail](../timing-byte.md) for the full
writeup — plus the two new findings from this batch:

- **Bits [3-0] — time value (`t`)**: `0`-`15`. `t=0` is a reserved fallback (~3s, confirmed
  separately in E9 05 testing). `t=15` combined with always=1 and scaler=1 appears to be a
  second reserved sentinel for genuine indefinite hold — see the finding above. Avoid both
  extremes unless you specifically want that reserved behavior.
- **Bits [5-4] — fade-out**: `00`=no fade, `01`=0.5s, `11`=1.5s all confirmed reliable.
  `10` is inconsistent (0.5s in one test, 1.0s in two others) — avoid relying on this value
  until it's retested in isolation.
- **Bit [6] — scaler**: `0` → `on_time ≈ 1.6×t`. `1` → `on_time = 3.0×t` (exact).
- **Bit [7] — extended-timeout mode**: `0` = normal. `1` combined with scaler=`0` →
  `on_time = 7.6×t` (confirmed, 6 data points). `1` combined with scaler=`1` → appears to be
  `on_time = 15×t` (one data point, `0xFE`), except at `t=15` which looks like true indefinite
  hold (`0xFF`) rather than computing to `15×15=225s`.

**Worked example — a mid-length ~15 second hold with a 0.5s fade, similar to `0x65` above:**
`t=5`, `fade_bits=01` (0.5s, avoiding the inconsistent `10` value), scaler=1 (`3.0×5=15.0s`
exact) → `(0<<7)|(1<<6)|(0b01<<4)|0b0101 = 0x55`

**Worked example — genuine indefinite hold (theory, not independently re-confirmed):**
`t=15`, always=1, scaler=1 → `(1<<7)|(1<<6)|(fade_bits<<4)|0b1111`. With no fade:
`0b11001111 = 0xCF`. This mirrors the structure of the confirmed `0xFF` example above (which
used fade_bits=`11`) — worth testing this specific byte if you need a reliable "stay lit
until told otherwise" packet, since it hasn't been isolated and retested on its own yet.

### `<MASK+COLOR>` (byte per LED zone)

`mask_color = (mask << 5) | color_index`, same as E9 05. Every packet in this batch uses
mask=`101` throughout (the standard "single color, all specified LEDs" value seen everywhere
else in this protocol) — the other mask values remain untested here too.

**Zone order to use when constructing a packet:** Center, Top-Right, Bottom-Right,
Bottom-Left, Top-Left — confirmed. Safe to rely on for zone-accurate WLED mapping.

**Worked example — matching the packet above showing all LEDs the same color:** all five
zones = color index 4 (blue in the 32-color palette), mask=`101` → `mask_color = 0xA4` for
every position. Matches `0x8301 E100 E909 00 23 0F A4 A4 A4 A4 A4 BD` exactly.

### `<VIBRATION>` (byte 14)

See [EMCOT.world vibration palette](https://emcot.world/Disney_MagicBand%2B_Bluetooth_Codes#Vibration_Palette)
— not yet independently confirmed against lab data.
